📖 User Manual & Operational Guide

  1. Paste your JWT: Copy the full token string (including all three dot-separated sections) from your application logs, browser devtools, or API response headers and paste it into the input.
  2. Instant decode: The token is parsed and decoded immediately. The color-coded display highlights the header (pink), payload (green), and signature (orange) sections.
  3. Review claims: The payload section shows all decoded claims with human-readable timestamps for exp, iat, and nbf fields.
  4. Check expiry: The status bar shows whether the token is currently valid, expired, or expiring soon (within 5 minutes).
  5. Copy output: Use the copy buttons to extract the decoded header or payload JSON for further processing.

🏗️ Architecture Insight: Why This Tool Runs Locally

JWT tokens frequently contain sensitive data: user IDs, roles, email addresses, session identifiers, and authorization scopes. Pasting a JWT token into an online decoder that makes a server request exposes this sensitive payload to a third party — a serious security risk in production environments.

This tool decodes JWT tokens using only JavaScript's built-in atob() function to Base64url-decode each section, and JSON.parse() to parse the JSON payload. No network request is ever made. The tool performs purely mathematical string operations in your browser's memory. Your token is never stored, logged, or transmitted anywhere.

❓ Frequently Asked Questions

Is it safe to decode a JWT token containing my auth credentials in a browser-based tool?
Yes — when using this specific tool, which is designed as a client-side-only debugger. Unlike online tools that make server requests, this decoder runs entirely in your browser's JavaScript engine. The token text you paste stays in browser RAM only. No network requests are made during decoding. You can verify this by opening browser DevTools → Network tab and confirming zero requests are made when you paste a token.
Can this tool verify the JWT signature?
No, and by design. Verifying a JWT signature requires the signing secret key (for HMAC algorithms like HS256) or the public key (for RSA/ECDSA algorithms). You should never paste your signing secret into any web tool. This tool only decodes the token structure for inspection purposes. Signature verification should be done server-side using your application's JWT library.
What JWT algorithms are supported for decoding?
All JWT algorithms can be decoded by this tool, since the header and payload sections are always Base64url-encoded JSON regardless of the signing algorithm. This includes HS256, HS384, HS512, RS256, RS384, RS512, ES256, ES384, ES512, PS256, and more. The algorithm is displayed from the alg claim in the decoded header.
Why does the "exp" claim show a timestamp far in the future or past?
JWT expiry (exp), issued-at (iat), and not-before (nbf) claims are stored as Unix timestamps (seconds since January 1, 1970 UTC). This tool converts them automatically to human-readable local datetime strings. If the timestamp appears incorrect, the issuing system may have set an unusual expiry duration, or the claim may be in milliseconds rather than seconds (a non-standard but common implementation).
