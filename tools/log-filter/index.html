📖 User Manual & Operational Guide

  1. Load your log: Click "Open File" to load a .log or .txt file, drop a file into the zone, or paste log lines directly and click "Parse Pasted Lines."
  2. Filter by level: Click "ERROR", "WARN", "INFO", or "DEBUG" to show only lines containing that log level keyword. Multiple selections are OR'd together. "All Levels" resets the filter.
  3. Grep search: Enter a plain text string or regex pattern to filter lines to those containing a match. Matches are highlighted in amber. The "Exclude" pattern removes matching lines.
  4. Export: Click "Export Filtered" to download the currently visible (filtered) log lines as a .txt file.

🏗️ Architecture Insight: Why This Tool Runs Locally

Production log files frequently contain sensitive operational data: user IDs, session tokens, IP addresses, database queries, and error stack traces. Uploading logs to a cloud service creates a significant privacy and compliance risk (GDPR, HIPAA, SOC2).

This tool reads log files using the browser's FileReader API, splits lines using JavaScript string operations, parses log levels via regex, and renders filtered lines to the DOM using a virtual render window that only builds DOM elements for visible lines. The entire log corpus stays in browser memory as a JavaScript string array — never transmitted anywhere.

❓ Frequently Asked Questions

What log formats does this parse filter large log files grep syntax browser tool support?
The tool supports any plain-text log format. It automatically detects common log level keywords (ERROR, FATAL, WARN, WARNING, INFO, DEBUG, TRACE) anywhere in each line, regardless of format. This covers standard formats including Log4j, Logback, Python logging, Winston (Node.js), Bunyan, syslog, NGINX access logs, Apache error logs, and custom application logs.
How large a log file can I analyze?
The tool has been tested with log files up to 100MB (millions of lines). Files are read into browser memory as a string, then split into a JavaScript array of line strings. The rendering uses DOM chunking to display a maximum of 2,000 lines at once to keep the UI responsive. For very large files (500MB+), browser memory limits may apply — the file loading may fail if the browser's available heap is insufficient.
Can I use regex syntax in the grep filter?
Yes — the grep input field accepts any valid JavaScript regular expression. For example, user_id=\d+ finds lines with numeric user IDs, Error|Exception finds error and exception lines, and ^\d{4}-\d{2}-\d{2} filters lines starting with a date. Invalid regex patterns are treated as literal string searches to prevent errors.
Can I filter log lines by a time range?
Use the grep search field with a timestamp pattern. For example, to see all logs from a specific hour, search for 2025-01-15 14: to filter lines from 2 PM on January 15. For Unix timestamp ranges, search for 173\d{7} to match a range of epoch timestamps. A dedicated time-range filter is a planned future feature.
